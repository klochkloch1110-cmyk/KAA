# AVL 84 - проектный каркас приложения v1

Документ собран на основе идеи из `ideya.md`. Это не "урезанный MVP", а каркас для полноценной версии v1, которую можно собирать поэтапно без расползания объема.

Важно: полный продукт все равно придется делать итерационно. Разница не в том, делать ли по шагам, а в том, что мы сразу фиксируем конечную целевую структуру системы и не строим времянку.

## 0. Короткая продуктовая рамка

### Цель продукта
Создать внутреннюю систему управления перевозками для компании АВЛ 84, которая:
- заменяет хаотичную работу через мессенджер и Excel;
- собирает заявки, рейсы, смены, документы и расходы в одной системе;
- упрощает контроль водителей, машин, пробега, заправок и ТО;
- автоматически формирует управленческие и клиентские отчеты;
- создает базу данных для последующего подключения AI-функций.

### Продуктовая позиция
Это не просто мобильное приложение для водителя. Это связка:
- мобильного приложения для водителя;
- кабинета руководителя для телефона, планшета и ПК;
- общей серверной логики, хранилища документов и отчетности.

## 2. Роли и права

Для вашей компании на старте достаточно 3 ролей. Не надо вводить сложную ролевую модель раньше времени.

## 2.1. Роли

### 1) Руководитель / Админ
Полный доступ ко всем данным и настройкам.

### 2) Водитель
Доступ только к своим рабочим данным, назначенной машине, своим заявкам, своим рейсам, своим сменам и общему чату.

### 3) Бухгалтер / Оператор (опционально, но лучше предусмотреть в модели)
Нужен на будущее. Может смотреть рейсы, документы, зарплатные и расходные отчеты, но без полного системного администрирования.

## 2.3. Правила доступа, которые нужно зафиксировать сразу
- водитель не видит чужие зарплаты;
- водитель не видит чужие документы и чужие смены;
- водитель видит только назначенную ему машину и связанную с ней рабочую информацию;
- руководитель видит все;
- оператор не должен иметь права удалять критичные данные;
- удаление заявок, рейсов и документов лучше заменить на мягкое удаление через статус `archived`.

## 3.2. Ключевые связи
- один водитель -> много смен;
- одна машина -> много смен;
- одна заявка -> много рейсов;
- одна смена -> много рейсов;
- один рейс -> много документов;
- одна машина -> много заправок, расходов, записей ТО;
- один водитель -> много начислений;
- чат может ссылаться на заявку, рейс, смену или машину.

## 3.3. Что важно заложить сразу
- все сущности должны иметь `created_at` и `updated_at`;
- вместо физического удаления для критичных данных лучше использовать статусы и архивирование;
- номера заявок и, при желании, рейсов должны генерироваться сервером;
- часть итоговых значений можно кэшировать в таблицах смен и заявок для скорости, но источником истины должны оставаться рейсы и документы;
- документы лучше хранить в storage, а в БД хранить только метаданные и связи.

## 4. Полный roadmap по спринтам

Ниже roadmap для полноценной v1. Предлагаю спринты по 2 недели. Итого 12 спринтов = 24 недели. Это реалистично для одного человека при плотной работе и моей постоянной поддержке.

## 5. Рекомендованный стек - окончательное решение

Ниже стек, который я рекомендую именно под ваши ограничения: маленький бюджет, самостоятельная сборка, необходимость мобилки и кабинета руководителя.

## 5.1. Основной выбор

### Клиент
- **Flutter**

Почему:
- одна кодовая база для Android и web;
- быстрый UI для форм, списков, загрузки фото;
- удобно собирать интерфейс водителя и руководителя в одной системе;
- меньше накладных расходов, чем при раздельных React Native + web.

### Backend и база данных
- **Supabase**

Почему:
- PostgreSQL как нормальная реляционная база;
- готовая аутентификация;
- storage для фото и файлов;
- realtime для чата и обновлений;
- row-level security для ролей;
- меньше инфраструктурной боли для соло-разработки.

### Карты
- **Yandex Maps API / Yandex MapKit**

Почему:
- локально релевантный сервис;
- нормальная работа по России;
- привычные маршруты;
- логичная интеграция с вашей бизнес-задачей.

### OCR
- **Yandex Vision OCR**

Почему:
- лучше бьется с русскоязычными документами и локальным стеком;
- удобно использовать для распознавания ТТН и базовых полей.

### Push-уведомления
- **Firebase Cloud Messaging**

Почему:
- стандартный и надежный канал для Android;
- нормально работает с Flutter.

### Логирование ошибок
- **Sentry**

Почему:
- критично для одиночной разработки;
- позволит видеть реальные ошибки на устройствах водителей.

### Генерация отчетов и фоновые задачи
- **Supabase Edge Functions**
- при необходимости позже: **небольшой Node.js service**

Почему:
- на старте можно не тащить полноценный отдельный backend;
- серверные функции хватит для генерации Excel, OCR-обработки, уведомлений и служебных задач.

## 5.2. Flutter-пакеты и технические опоры

Рекомендуемый практичный набор:
- state management: **Riverpod**;
- routing: **go_router**;
- network/storage bindings: официальный **supabase_flutter**;
- serialization: **freezed** + **json_serializable**;
- forms: **flutter_form_builder** или аккуратные собственные формы;
- tables/admin UI: DataTable / PlutoGrid для web-части;
- camera/upload: `image_picker`, при необходимости `camera`;
- maps: пакет под Yandex Maps по выбранной платформе;
- local cache: `hive` или `isar` только если реально потребуется офлайн-буфер.

## 5.3. Что я не рекомендую на старте
- не рекомендую сразу поднимать тяжелый кастомный backend на NestJS/Java/.NET, если вы делаете продукт сами;
- не рекомендую отдельные кодовые базы для мобильного приложения и web-кабинета без сильной причины;
- не рекомендую микросервисы;
- не рекомендую сложный offline-first до тех пор, пока не станет ясно, что сеть реально часто рвется;
- не рекомендую строить AI-архитектуру до стабилизации первичных данных.

## 5.4. Окончательный стек в одной строке

**Flutter + Supabase + PostgreSQL + Supabase Storage + Supabase Realtime + Yandex Maps + Yandex OCR + Firebase Cloud Messaging + Sentry**

Это самый здравый баланс между скоростью разработки, стоимостью владения и масштабируемостью для вашего случая.
