# AVL 84 - комплексный план разработки v1

Документ фиксирует переход от идеи, ТЗ и Figma-прототипа к инженерной реализации.

Главная задача ближайшего этапа - не построить сразу всю большую v1, а собрать рабочее операционное ядро: заявка -> рейс -> фото ТТН -> смена -> отчет.

---

## 1. Текущее состояние

Проект уже содержит:

- исходное описание бизнес-процесса в `ideya.md`;
- проектный каркас v1 в `app_blueprint_v1.md`;
- ТЗ по экранам и сценариям;
- статусы и бизнес-правила;
- ER-модель PostgreSQL/Supabase;
- стартовую архитектуру Flutter/Supabase;
- UI/UX-ТЗ;
- интерактивный React/Vite-прототип из Figma Make в `prototypes/figma-make`.

---

## 2. Зафиксированное техническое решение

Основной стек разработки:

- Flutter для Android-приложения водителя;
- Flutter Web для кабинета руководителя;
- Supabase Auth для входа и ролей;
- Supabase Postgres для данных;
- Supabase Storage для фото ТТН, чеков и документов;
- Supabase Edge Functions для серверных операций;
- Yandex Maps / Yandex MapKit для маршрутов;
- Yandex OCR для распознавания ТТН на следующем этапе;
- Firebase Cloud Messaging для push-уведомлений на следующем этапе;
- Sentry для мониторинга ошибок.

Figma Make-прототип используется как визуальный и сценарный референс, но не как финальная кодовая база.

---

## 3. Цель первого рабочего релиза

Первый рабочий релиз должен заменить самую болезненную ручную часть процесса:

1. Руководитель создает заявку.
2. Назначает водителя и машину.
3. Водитель видит свою заявку.
4. Водитель открывает маршрут.
5. После рейса водитель отправляет отчет с номером ТТН, объемом и фото.
6. В конце дня водитель закрывает смену.
7. Руководитель видит рейсы, документы и смены.
8. Руководитель выгружает простую таблицу по рейсам.

---

## 4. Scope первого рабочего релиза

### 4.1. Входит

#### Роли и вход
- вход по email/паролю или логину/паролю через Supabase Auth;
- роли `admin`, `driver`, `operator` в профиле пользователя;
- разделение интерфейса по роли;
- блокировка пользователя через статус.

#### Руководитель
- базовый dashboard;
- список водителей;
- создание/редактирование водителя;
- список машин;
- создание/редактирование машины;
- справочники: заказчики, организации, материалы, локации;
- список заявок;
- создание заявки;
- назначение водителя и машины;
- журнал рейсов;
- просмотр карточки рейса и фото ТТН;
- журнал смен;
- просмотр карточки смены;
- простая выгрузка рейсов за период.

#### Водитель
- главный экран;
- список своих заявок;
- карточка заявки;
- открытие маршрута внешней ссылкой;
- создание рейсового отчета;
- загрузка фото ТТН и сопроводительных документов;
- список рейсов текущей смены;
- закрытие смены;
- профиль.

#### Данные и файлы
- хранение основных сущностей в PostgreSQL;
- хранение документов в Supabase Storage;
- метаданные документов в таблице `documents`;
- базовый audit log для критичных действий.

### 4.2. Не входит в первый рабочий релиз

- реальный OCR;
- полноценный чат;
- push-уведомления;
- сложная аналитика;
- зарплатные ведомости;
- расходы, заправки и ТО в полном объеме;
- AI-агент;
- сложная offline-first синхронизация;
- iOS-сборка.

Эти модули заложены в структуре, но подключаются после стабилизации операционного ядра.

---

## 5. Архитектурные принципы реализации

1. Источник истины - БД, а не чат или локальное состояние клиента.
2. Водитель не должен видеть чужие заявки, рейсы, смены, документы и зарплаты.
3. Права должны защищаться на уровне Supabase RLS, а не только в UI.
4. Критичные данные не удаляются физически из интерфейса: используется статус или архивирование.
5. Фото документов хранятся в Storage, в БД лежат метаданные и связи.
6. Серверная логика нужна для нумерации заявок, отчетов, OCR и уведомлений.
7. UI водителя должен быть проще и крупнее UI руководителя.
8. Прототип Figma Make нельзя переносить напрямую без адаптации архитектуры.

## 6. Backlog ближайших 6 недель

## Неделя 1. Supabase-фундамент

Цель: подготовить базу данных, роли, storage и правила доступа.

Задачи:
- создать структуру `supabase/`;
- подготовить миграцию ядра БД;
- добавить enum-типы и check-ограничения;
- добавить индексы;
- включить RLS;
- подготовить черновые политики;
- описать storage buckets;
- подготовить seed-данные.

Критерий готовности:
- миграции можно применить к Supabase-проекту;
- таблицы ядра созданы;
- политики не дают водителю читать чужие данные;
- есть понятная структура хранения документов.

## Неделя 2. Flutter-каркас

Цель: создать основу приложения.

Задачи:
- создать Flutter-проект в `app/`;
- подключить `supabase_flutter`;
- подключить `flutter_riverpod`;
- подключить `go_router`;
- сделать auth flow;
- сделать разделение `/admin` и `/driver`;
- подготовить тему по мотивам Figma Make;
- создать общие виджеты: кнопки, поля, карточки, статусы, empty/error/loading.

Критерий готовности:
- пользователь входит;
- после входа попадает в правильную область по роли;
- есть базовый shell админа и водителя.

## Неделя 3. Справочники, машины, водители

Цель: наполнить систему управляемыми данными.

Задачи:
- CRUD водителей;
- CRUD машин;
- CRUD заказчиков;
- CRUD организаций;
- CRUD материалов;
- CRUD локаций;
- карточка водителя;
- карточка машины.

Критерий готовности:
- админ может подготовить данные для создания заявки;
- водитель не может редактировать справочники и чужие данные.

## Неделя 4. Заявки

Цель: создать полный цикл заявки до назначения водителю.

Задачи:
- список заявок админа;
- форма создания заявки;
- редактирование заявки;
- генерация номера заявки;
- назначение водителя и машины;
- карточка заявки админа;
- список заявок водителя;
- карточка заявки водителя;
- кнопка открытия маршрута.

Критерий готовности:
- админ создает заявку;
- водитель видит только свои назначенные заявки;
- водитель может открыть маршрут.

## Неделя 5. Рейсы и документы

Цель: перевести рейсовые отчеты из мессенджера в систему.

Задачи:
- форма рейсового отчета водителя;
- ручной ввод ТТН и объема;
- загрузка фото ТТН;
- загрузка сопроводительных фото;
- запись документов в `documents`;
- журнал рейсов админа;
- карточка рейса с просмотром фото;
- базовые статусы рейса и OCR `not_required` / `pending`.

Критерий готовности:
- водитель отправляет рейс с фото;
- админ видит рейс и документ;
- документ можно открыть для проверки.

## Неделя 6. Смены и первая выгрузка

Цель: закрыть дневной цикл работы.

Задачи:
- открытие смены;
- закрытие смены;
- автоматический подсчет рейсов и объема;
- ручной ввод пробега и заправки;
- журнал смен админа;
- карточка смены;
- простая выгрузка CSV/XLSX по рейсам за период;
- ручной тест на сценарии рабочего дня.

Критерий готовности:
- рабочий день можно провести через систему от заявки до отчета;
- руководитель может получить первичную таблицу по рейсам.

## 8. Критические решения, которые нужно подтвердить перед кодингом

1. Водители на первом этапе работают только на Android.
2. Основной стек остается Flutter + Supabase.
3. Первый отчет - по рейсам за период.
4. OCR подключается после того, как стабильно работают рейсы и фото.
5. Чат не блокирует запуск первого рабочего релиза.
6. Реальные Excel-шаблоны заказчиков нужно добавить в проект до разработки финальных выгрузок.

---

## 9. Definition of Done для первого рабочего релиза

Первый рабочий релиз считается готовым, если:

- админ может создать водителя, машину и заявку;
- заявка назначается водителю и машине;
- водитель видит назначенную заявку;
- водитель может отправить рейсовый отчет с фото ТТН;
- водитель может закрыть смену;
- админ видит рейсы, смены и документы;
- данные защищены RLS;
- фото хранятся в Storage;
- простая выгрузка по рейсам работает;
- сценарий проверен на тестовых данных, близких к реальному рабочему дню.
