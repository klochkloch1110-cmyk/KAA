# AVL 84 - что нужно от владельца проекта для интеграции Supabase

Этот список нужен, чтобы перейти от dev-контура к реальной проверке backend, RLS и Storage.

## 1. Supabase-проект

Нужно предоставить разработчику или внести самостоятельно:

- `SUPABASE_URL` - URL проекта вида `https://xxxxx.supabase.co`;
- `SUPABASE_ANON_KEY` - публичный anon key для Flutter-клиента;
- доступ к Supabase SQL Editor или возможность применить SQL-файлы из `supabase/migrations/`;
- подтверждение, что bucket `documents` создан миграцией и остается приватным.

Нельзя передавать и коммитить:

- service role key;
- пароли реальных пользователей;
- production-персональные данные.

## 2. Тестовые пользователи Auth

В Supabase Auth нужно создать 3 тестовых аккаунта и передать их `auth.users.id`:

| Роль | Рекомендуемый email | Что нужно передать |
|---|---|---|
| Админ | `admin@avl84.local` | UUID из `auth.users.id` |
| Водитель 1 | `driver1@avl84.local` | UUID из `auth.users.id` |
| Водитель 2 | `driver2@avl84.local` | UUID из `auth.users.id` |

Эти UUID нужны для временной замены placeholder-значений в:

```text
supabase/seed/001_dev_seed.sql
supabase/tests/rls_smoke_checks.sql
```

## 3. Кто применяет миграции

Нужно выбрать вариант:

1. Пользователь применяет SQL сам через Supabase SQL Editor.
2. Пользователь устанавливает Supabase CLI и дает разработчику работать локально.
3. Пользователь временно дает доступ к Supabase проекту без передачи service-role key в репозиторий.

Рекомендуемый безопасный вариант на старте: применить SQL через Supabase SQL Editor вручную.

## 4. Данные для ручной бизнес-проверки

Для проверки первого рабочего дня нужны реальные или тестовые значения:

- 1-2 реальные машины: марка, модель, госномер, текущий пробег;
- 1-2 водителя: ФИО, телефон;
- 1 заказчик;
- 1 материал;
- точка погрузки;
- точка разгрузки;
- ставка водителя за рейс;
- управленческая ставка за тонну/м3;
- тестовое фото или PDF ТТН.

## 5. Что будет проверяться после получения данных

1. Вход админа и водителя через Supabase Auth.
2. Разделение интерфейса по ролям.
3. Создание/назначение заявки.
4. Открытие смены водителем.
5. Отправка рейсового отчета с фото ТТН.
6. Загрузка файла в Storage.
7. Запись метаданных в `documents`.
8. Открытие ТТН админом через signed URL.
9. Закрытие смены.
10. CSV-выгрузка.
11. RLS smoke-check: водитель не видит чужие рейсы, смены и документы.
