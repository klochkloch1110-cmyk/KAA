# AVL 84 - чеклист приемки первого рабочего релиза 0.1

Чеклист нужен для ручной проверки перед первой эксплуатацией на реальных сменах.

---

## 1. Авторизация и роли

- [ ] Админ может войти.
- [ ] Водитель может войти.
- [ ] Заблокированный пользователь не может войти.
- [ ] Админ попадает в кабинет руководителя.
- [ ] Водитель попадает в водительский интерфейс.
- [ ] Водитель не может открыть админские маршруты.

---

## 2. Справочники

- [ ] Админ может создать заказчика.
- [ ] Админ может создать организацию-отправителя.
- [ ] Админ может создать организацию-получателя.
- [ ] Админ может создать материал.
- [ ] Админ может создать локацию.
- [ ] Водитель не может редактировать справочники.

---

## 3. Машины и водители

- [ ] Админ может создать водителя.
- [ ] Админ может создать машину.
- [ ] Госномер машины уникален.
- [ ] Нельзя назначить машину в статусе `repair`, `service`, `inactive` на новую заявку.
- [ ] Водитель видит только назначенную ему машину в рабочем контексте.

---

## 4. Заявки

- [ ] Админ может создать заявку.
- [ ] Номер заявки генерируется корректно.
- [ ] Админ может назначить водителя и машину.
- [ ] Назначенная заявка появляется у водителя.
- [ ] Водитель не видит чужие заявки.
- [ ] Отмененная заявка недоступна для создания рейса.

---

## 5. Рейсы и документы

- [ ] Водитель может создать рейсовый отчет.
- [ ] Номер ТТН обязателен.
- [ ] Объем обязателен.
- [ ] Фото ТТН обязательно.
- [x] Фото загружается в Storage. _Реализовано для Supabase-контура через bucket `documents`, требует проверки на живом проекте._
- [x] Метаданные фото появляются в `documents`. _Реализована запись `entity_type=trip`, `document_type=ttn`, `file_path`, `metadata_json`._
- [ ] Админ видит рейс в журнале.
- [x] Админ может открыть фото ТТН. _Через signed URL из карточки документа._
- [x] Водитель не видит документы чужих рейсов. _Зафиксировано RLS для `documents` и Storage policies; требуется интеграционная проверка в Supabase._

---

## 6. Смены

- [ ] Водитель может открыть смену.
- [ ] У водителя нет двух открытых смен без специального разрешения.
- [ ] Водитель может закрыть смену.
- [x] Пробег обязателен.
- [x] Пробег не может быть меньше последнего подтвержденного пробега машины.
- [x] Количество рейсов считается автоматически.
- [x] Общий объем считается автоматически.
- [ ] Админ видит смену в журнале.

---

## 7. Выгрузка

- [x] Админ может выбрать период.
- [x] Админ может сформировать таблицу рейсов.
- [x] В выгрузке есть дата, заявка, водитель, машина, ТТН, объем, статус.
- [x] Выгрузка совпадает с данными в интерфейсе.

---

## 8. Безопасность данных

- [x] RLS включен на таблицах ядра. _В миграции включен RLS; требуется проверка примененной БД._
- [x] Водитель не может получить чужие рейсы прямым запросом. _RLS описан в миграции; требуется интеграционный тест._
- [x] Водитель не может получить чужие смены прямым запросом. _RLS описан в миграции; требуется интеграционный тест._
- [x] Водитель не может получить чужие документы прямым запросом. _RLS и Storage policies описаны в миграциях; требуется интеграционный тест._
- [x] Админ видит все операционные данные. _RLS для admin/operator разрешает чтение операционных данных; требуется интеграционный тест._
- [ ] Критичные изменения попадают в audit log.

---

## 9. UX-проверка на реальном сценарии

- [ ] Водитель понимает следующий шаг на главном экране.
- [ ] Отчет по рейсу можно отправить быстро.
- [ ] Ошибки формы понятны без технических терминов.
- [ ] Фото можно переснять/заменить до отправки.
- [ ] Админ находит нужный документ за 2-3 действия.
