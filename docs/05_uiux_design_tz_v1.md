## 2. Цель дизайн-задачи

Подготовить Figma-пакет, по которому можно:
- быстро согласовать логику продукта;
- проверить основные сценарии пользователя;
- избежать переделок на этапе разработки;
- передать разработчику понятные wireframes, flow и UI-спецификацию.

Итог работы дизайнера должен позволить перейти к разработке без расплывчатых трактовок типа "тут потом решим".

---

## 3. Продуктовый контекст

### 3.1. Что это за продукт
Внутренняя система управления перевозками для небольшой компании с собственными самосвалами.

### 3.2. Что система должна заменить
- хаотичную коммуникацию в мессенджере;
- ручной сбор фото ТТН;
- ручную сводку рейсов и смен;
- ручной контроль пробега, заправок, ТО и расходов;
- ручную подготовку отчетов для клиента и внутренних таблиц.

### 3.3. Главная ценность интерфейса
Интерфейс должен не впечатлять декоративностью, а:
- ускорять операционные действия;
- уменьшать ошибки ввода;
- показывать текущий статус работы;
- давать руководителю контроль над ситуацией в 1-2 экрана.

## 4. Роли пользователей

## 4.1. Водитель

### Цели пользователя
- быстро понять, что назначено;
- открыть маршрут;
- отправить отчет по рейсу;
- закрыть смену;
- увидеть свои рейсы и заработок;
- общаться с руководителем и коллегами по работе.

### UX-риски
- низкое внимание;
- работа в полевых условиях;
- необходимость быстрого ввода;
- возможные проблемы со связью;
- работа с фото и документами на ходу между задачами.

### Дизайн-вывод
Водительский интерфейс должен быть простым, крупным, очень понятным и с минимальным количеством решений на экране.

## 4.2. Руководитель / Админ

### Цели пользователя
- создавать и назначать заявки;
- контролировать рейсы и смены;
- видеть документы и OCR-расхождения;
- отслеживать машины, расходы и ТО;
- считать начисления;
- собирать отчеты и выгрузки.

### UX-риски
- большой объем данных;
- высокая плотность таблиц;
- необходимость быстрого поиска и фильтрации;
- необходимость работать и с телефона, и с широкого экрана.

### Дизайн-вывод
Админ-интерфейс должен быть компактным, табличным, управляемым через фильтры и не должен терять читаемость при большом количестве данных.

## 4.3. Оператор
Роль на будущее. В дизайне не требуется отдельная ветка экранов. Используются экраны админа, но с частично скрытыми действиями.

## 6. Границы дизайн-задачи

### Входит в задачу
- user flows;
- low-fidelity wireframes;
- mid/high-fidelity UI после согласования логики;
- интерактивные прототипы основных сценариев;
- базовая дизайн-система;
- спецификация состояний;
- адаптивные правила для mobile/tablet/desktop.

### Не входит в первую итерацию
- промо-сайт;
- маркетинговые экраны;
- сложная бренд-платформа;
- сверхдетальная анимационная система;
- геймификация уровня отдельного продукта.

Важно: дизайнеру не нужно выдумывать второстепенные "вау-фичи". Основная задача - рабочий, ясный и масштабируемый интерфейс.

## 7. Дизайн-принципы

## 7.1. Общие
- функциональность важнее декоративности;
- статус важнее украшения;
- действие важнее текста-описания;
- одна сущность = один главный сценарий;
- минимум лишних полей на экране;
- все критичные состояния должны быть визуально различимы.

## 7.2. Для водителя
- один главный CTA на экран;
- минимум конкурирующих элементов;
- акцент на текущую смену и текущую заявку;
- быстрый доступ к камере и маршруту;
- читаемые статусы по рейсу и OCR.

## 7.3. Для админа
- плотные, но не перегруженные таблицы;
- удобная фильтрация;
- быстрый доступ к документам;
- понятная иерархия статусов;
- системный подход к спискам, карточкам и боковым панелям.

---

## 8. Тональность и визуальное направление

### Базовый стиль
- современный, чистый, сдержанный;
- индустриальный, но не грубый;
- деловой интерфейс, а не "развлекательное приложение";
- нейтральная палитра с одним-двумя акцентными цветами;
- хороший контраст и читаемость в уличных условиях.

### Не делать
- кислотные цвета;
- тяжелые градиенты;
- перегруженные карточки;
- декоративные иллюстрации без практической пользы;
- слишком "игровой" язык интерфейса.

### Допустимая эмоциональность
На стороне водителя можно аккуратно показать личный прогресс и заработок, но не превращать интерфейс в мотивационную ленту.

## 10. Основные user flows, которые нужно собрать в первую очередь

## 10.1. Водитель
1. Логин -> главный экран.
2. Главный экран -> список заявок -> карточка заявки -> открыть маршрут.
3. Карточка заявки -> создать рейсовый отчет -> приложить ТТН -> отправить.
4. Главный экран -> текущая смена -> закрытие смены -> отправка сменного отчета.
5. Главный экран -> машина -> просмотр карточки машины.
6. Главный экран -> чат -> ответ на сообщение.
7. Главный экран -> история смен -> просмотр начислений.

## 10.2. Руководитель
1. Логин -> dashboard.
2. Dashboard -> список заявок -> создание заявки -> назначение водителя/машины.
3. Dashboard -> журнал рейсов -> карточка рейса -> проверка ТТН и OCR.
4. Dashboard -> журнал смен -> карточка смены -> подтверждение.
5. Dashboard -> документы -> фильтр -> просмотр -> скачивание.
6. Dashboard -> машины -> карточка машины -> ТО/заправки/расходы.
7. Dashboard -> зарплата -> период -> детализация -> выгрузка.
8. Dashboard -> отчеты -> выбор параметров -> preview -> export.

## 13. Детализация ключевых экранов для дизайнера

Ниже не повторяются все бизнес-поля из ТЗ по экранам. Здесь акцент именно на композиции, приоритетах и поведении UI.

## 13.1. Главный экран водителя

### Цель экрана
За 3-5 секунд дать ответ на вопросы:
- есть ли активная смена;
- какие заявки сейчас актуальны;
- что нужно сделать следующим действием;
- сдал ли водитель рейсовые отчеты;
- нужно ли закрыть смену.

### Приоритет блоков сверху вниз
1. Статус смены.
2. Главный CTA.
3. Активные заявки.
4. Сводка за сегодня.
5. Машина.
6. Последние уведомления.

### Главный CTA в зависимости от контекста
- если нет открытой смены: `Открыть смену` или `Перейти к заявкам`;
- если есть активная заявка: `Создать отчет по рейсу`;
- если все рейсы сданы: `Закрыть смену`.

### UI-требования
- верхний блок должен визуально показывать статус смены;
- CTA всегда должен быть заметнее вторичных действий;
- заявки отображать карточками с ключевой информацией без перегруза;
- номера ТТН, уже отправленные по смене, показывать коротким списком или компактными чипами.

## 13.2. Карточка заявки водителя

### Приоритет данных
1. Маршрут и точки A/B.
2. Материал и объем.
3. Ставка за рейс.
4. Примечание.
5. История уже отправленных рейсов.

### UI-требования
- карта или блок маршрута должен быть визуально выше второстепенной информации;
- кнопка `Открыть маршрут` - первичная;
- кнопка `Отчитаться по рейсу` - рядом или сразу под маршрутом;
- история рейсов не должна перетягивать внимание на себя.

## 13.3. Форма рейсового отчета

### Цель
Минимизировать ошибки и количество действий.

### Компоновка
1. Автозаполненный summary по заявке.
2. Поле ТТН.
3. Поле объема.
4. Фото ТТН.
5. Сопроводительные документы.
6. Комментарий.
7. Основная кнопка отправки.

### UX-правила
- фото ТТН должно быть явно обязательным;
- после добавления фото нужен крупный preview;
- добавить явный статус "OCR проверит документ после отправки";
- если фото не добавлено, ошибка должна быть заметной и локальной;
- не прятать основное действие слишком низко без фиксированной кнопки.

### Полезные детали
- предусмотреть состояние повторной съемки;
- предусмотреть множественные сопроводительные документы;
- предусмотреть подсказку по качеству фото: не мять, не обрезать номер, достаточный свет.

## 13.4. Экран закрытия смены

### Цель
Подтвердить итог, а не заставить водителя вручную пересобирать день.

### Компоновка
1. Summary по смене.
2. Разбивка по заявкам.
3. Поле пробега.
4. Поле заправки.
5. Комментарий.
6. CTA отправки.

### UI-требования
- авторассчитанные показатели должны быть визуально отделены от ручного ввода;
- если есть незавершенные рейсы, показать предупреждающий блок выше CTA;
- поле пробега должно быть максимально понятным и сопровождаться последним известным значением.

## 13.5. Dashboard админа

### Цель
Дать обзор текущей операционной ситуации за один экран.

### Обязательные виджеты
- активные заявки;
- водители на смене;
- машины на линии;
- количество рейсов за сегодня;
- OCR-проблемы;
- незакрытые смены;
- расходы за период;
- быстрые действия.

### UI-требования
- не превращать дашборд в свалку карточек;
- сверху должен быть уровень KPI/summary;
- ниже - проблемные и оперативные списки;
- таблицы последних рейсов и смен не должны быть слишком длинными на первом экране.

## 13.6. Список заявок админа

### Цель
Быстро находить, фильтровать и управлять заявками.

### UI-требования
- таблица должна поддерживать status badges;
- фильтры должны быть быстрыми, не глубоко спрятанными;
- массовые действия можно не проектировать на старте, если они не подтверждены бизнесом;
- кнопка `Создать заявку` должна быть всегда заметна.

## 13.7. Карточка рейса и OCR mismatch

### Цель
Позволить руководителю быстро сверить данные.

### Компоновка
1. Header с ключевыми метаданными.
2. Блок ручного ввода.
3. Блок OCR-результата.
4. Фото ТТН.
5. Сопроводительные документы.
6. История статусов и комментарии.
7. Кнопки действия.

### UI-требования
- визуально различать "введено водителем" и "распознано OCR";
- mismatch должен быть заметен без чтения длинных текстов;
- фото ТТН должно открываться крупно и удобно для ручной сверки.

## 13.8. Архив документов

### Цель
Найти нужный документ максимум в 2-3 действия.

### UI-требования
- на desktop использовать split-view: список + preview;
- фильтры должны включать период, тип, водителя, машину, заявку, заказчика, OCR-статус;
- preview должен поддерживать zoom и открытие в full view;
- нужно предусмотреть пакетную выгрузку, даже если визуально это одна кнопка без сложного сценария.

## 14. Состояния, которые обязательно нужно отрисовать

Без этого дизайн будет неполным.

## 14.1. Общие
- loading;
- empty state;
- error state;
- success state;
- offline / проблемы сети;
- blocked / restricted state;
- confirmation modal;
- destructive action confirmation;
- archived state.

## 14.2. Для водителя
- нет активной смены;
- нет назначенных заявок;
- рейс отправлен;
- рейс требует проверки;
- смена не может быть закрыта без пробега;
- OCR mismatch отмечен руководителем;
- не загружено фото ТТН.

## 14.3. Для админа
- пустой список заявок;
- пустой журнал рейсов;
- нет документов по фильтру;
- OCR queue empty;
- заявка без назначенного водителя;
- машина недоступна для назначения;
- предупреждение о спорном пробеге;
- архивный объект только для чтения.

---

## 15. Статусы и их визуализация

Дизайнер должен сразу заложить универсальную статусную систему.

### Статусы заявок
- draft
- assigned
- in_progress
- completed
- cancelled
- archived

### Статусы рейсов
- submitted
- verified
- needs_review
- rejected

### Статусы OCR
- pending
- matched
- mismatch
- failed
- not_required

### Статусы смен
- open
- submitted
- approved
- needs_review

### Статусы машин
- active
- service
- repair
- inactive

### Требование к дизайну статусов
- статусы должны различаться не только цветом, но и текстом/иконкой;
- warning и error нельзя делать визуально одинаковыми;
- archived и inactive не должны путаться с completed;
- статусные токены должны быть едиными для всех экранов.

---

## 16. Требования к формам

## 16.1. Общие
- единый паттерн лейблов, подсказок и ошибок;
- обязательные поля помечать явно;
- ошибки показывать рядом с полем;
- длинные формы разбивать на логические секции;
- primary action должен быть предсказуемым по расположению.

## 16.2. Формы водителя
- максимум автозаполнения;
- минимум необязательных полей на первом экране;
- фото и документы не должны выглядеть второстепенно;
- числовые поля проектировать под мобильную клавиатуру.

## 16.3. Формы админа
- справочники должны иметь pattern выбора из списка + возможность добавить новую сущность, если это согласовано;
- фильтры в таблицах должны быть быстрыми и многоразовыми;
- в длинных формах заявки допустим двухколоночный layout на desktop.

---

## 17. Требования к таблицам и спискам

## 17.1. Таблицы админа
- sticky header желателен;
- сортировка по ключевым колонкам;
- понятные пустые состояния;
- status badges;
- быстрый переход в карточку по клику на строку;
- действия в строке не должны перегружать таблицу.

## 17.2. Списки водителя
- карточки вместо тяжелых таблиц;
- на карточке 4-6 ключевых полей максимум;
- главное действие должно быть видно сразу.

---

## 18. Требования к карте и маршруту

### Для дизайна нужно подготовить
- карточку точки A;
- карточку точки B;
- блок маршрута в заявке;
- CTA `Открыть в Яндекс Картах` или аналогичный;
- вариант без встроенной карты, если карта загружается медленно;
- fallback-состояние без координат.

### Важно
Не завязывать критичный сценарий только на красиво встроенную карту. Пользователю всегда нужен простой способ открыть маршрут внешне.

---

## 19. Требования к чату

### Что нужно в v1
- общий рабочий чат;
- текст;
- ответ на сообщение;
- фото-вложения;
- системные сообщения;
- ссылки на сущности.

### Что дизайнеру не нужно усложнять
- реакциями;
- сторис-подобными паттернами;
- сложными профилями внутри чата;
- декоративными экспериментами.

### UI-требования
- сообщения должны быть читабельны при длинной переписке;
- системные сообщения должны отличаться от пользовательских;
- вложения должны открываться в preview;
- reply-паттерн должен быть заметен, но не занимать полэкрана.

## 20. Требования к дизайн-системе

## 20.2. Компоненты
- buttons: primary, secondary, ghost, destructive;

- search field;
- segmented control;

- chips/tags;
- badges/statuses;
- cards;

## 20.3. Компонентные состояния

- loading, где применимо.

## 21. Контент и тексты интерфейса

### Требования
- язык интерфейса: русский;
- короткие и прямые CTA;
- без канцелярита и слишком длинных предупреждений;
- статусы и ошибки должны быть понятны без технических терминов.

### Примеры правильного тона
- `Отправить отчет`
- `Добавить фото ТТН`
- `Нужна проверка`
- `Смена не закрыта`
- `Пробег меньше последнего сохраненного`

### Примеры плохого тона
- `Произошла ошибка выполнения операции`
- `Действие не может быть произведено`
- `Неверный формат данных верификации`

---

## 22. Accessibility и usability

Минимальные требования:
- достаточный контраст;
- кликабельные зоны не слишком мелкие;
- понятные фокус-состояния для web;
- цвет не должен быть единственным носителем смысла;
- ключевые действия должны читаться на солнце и на дешевых экранах Android.

Это особенно важно для водителей. Красивый, но слабоконтрастный интерфейс здесь просто плохой.

## 23. Требования к интерактивному прототипу

Дизайнер должен собрать минимум 2 полноценных кликабельных сценария.

## 23.1. Прототип водителя
Обязательный сценарий:
1. Логин.
2. Попадание на главный экран.
3. Открытие заявки.
4. Открытие маршрута.
5. Заполнение рейсового отчета.
6. Добавление фото.
7. Отправка отчета.
8. Переход в список рейсов.
9. Закрытие смены.

## 23.2. Прототип админа
Обязательный сценарий:
1. Логин.
2. Dashboard.
3. Создание заявки.
4. Назначение водителя и машины.
5. Переход в журнал рейсов.
6. Открытие карточки спорного рейса.
7. Просмотр OCR mismatch.
8. Переход в архив документов.
9. Формирование отчета.

### Что должно быть кликабельно
- основная навигация;
- ключевые CTA;
- переходы между списком и карточкой;
- просмотр модальных окон;
- preview документа;
- фильтры в базовом объеме.

## 27. Критерии приемки UI/UX-документации

Работу можно считать достаточной для старта разработки, если:
- все ключевые сценарии покрыты;
- по каждому важному экрану есть состояние по умолчанию и исключения;
- навигация водителя и админа понятна;
- формы не противоречат бизнес-логике;
- статусы визуально консистентны;
- документы, OCR и отчетность не потеряны в UX;
- админка продумана для desktop/tablet;
- разработчик может по макетам понять структуру без дополнительных догадок.

## 28. Особые замечания для дизайнера

1. Не пытаться сделать интерфейс "слишком модным" в ущерб скорости работы.
2. Не прятать критичные действия в многослойные меню.
3. Не строить главный экран водителя как информационную свалку.
4. Не делать дашборд админа из 20 декоративных карточек без иерархии.
5. Не относиться к документам как к второстепенному модулю - это одно из ядер продукта.
6. Не забывать, что OCR - это подсказка для контроля, а не блокирующий механизм.
