# AVL 84 - стартовая архитектура Flutter/Supabase проекта v1

Документ описывает, как лучше организовать проект, чтобы его реально можно было собирать одному человеку без лишнего технического веса.

---

## 1. Архитектурный принцип

Для этого проекта нужен не "идеальный энтерпрайз", а дисциплинированная, понятная и расширяемая структура.

Правильный базовый подход:
- один Flutter-проект;
- Android как основной мобильный клиент;
- Flutter Web для кабинета руководителя;
- Supabase как backend-платформа;
- feature-first структура с умеренным разделением на слои.

Не нужно на старте делать:
- отдельный фронтенд для web на React;
- тяжелый кастомный backend;
- микросервисы;
- отдельный BFF;
- overengineering ради архитектурной красоты.

## 3. Формат репозитория

На старте достаточно одного репозитория:

```text
avl84_app/
  app/
  supabase/
  docs/
```

### Что где лежит
- `app/` - Flutter-клиент;
- `supabase/` - SQL-миграции, edge functions, storage notes, seed data;
- `docs/` - документация, ТЗ, схемы, правила.

Это проще, чем сразу городить монорепо из нескольких приложений.

## 5. Как организовать feature-модули

Для каждого большого модуля:

```text
features/orders/
  data/
    datasources/
    dto/
    repositories/
  domain/
    entities/
    repositories/
    usecases/
  presentation/
    controllers/
    screens/
    widgets/
```

### Почему так
- модуль собран в одном месте;
- UI, данные и логика не размазаны по всему проекту;
- легче поддерживать соло-разработчику;
- можно не делать фанатичный clean architecture, но сохранить порядок.

Если модуль маленький, его можно упрощать. Не надо копировать эту вложенность туда, где достаточно пары файлов.

## 6. Слои ответственности

## 6.1. Presentation
Содержит:
- экраны;
- виджеты;
- контроллеры состояния;
- локальную UI-валидацию.

Не должен содержать:
- прямую SQL-логику;
- размазанную бизнес-логику;
- работу со storage напрямую без абстракции.

## 6.2. Domain
Содержит:
- бизнес-сущности;
- use cases;
- интерфейсы репозиториев;
- чистые правила преобразования.

## 6.3. Data
Содержит:
- доступ к Supabase;
- DTO;
- мапперы;
- репозитории;
- работу с edge functions;
- загрузку файлов.

---

## 7. State management

Рекомендую **Riverpod**.

### Почему
- хорошо масштабируется;
- не заставляет писать лишний бойлерплейт;
- удобно комбинируется с async-состояниями;
- подходит и для mobile, и для web.

### Подход
- `StateNotifier` / `Notifier` / `AsyncNotifier` использовать по ситуации;
- короткоживущие состояния форм держать рядом с экраном;
- глобальные состояния: auth, профиль, выбранный период, уведомления.

---

## 8. Навигация

Рекомендую **go_router**.

### Основные навигационные ветки
- auth;
- driver area;
- admin area.

### Пример маршрутов
```text
/login
/driver/home
/driver/orders
/driver/orders/:id
/driver/trips/new
/driver/shift/current
/driver/chat
/admin/dashboard
/admin/orders
/admin/orders/new
/admin/orders/:id
/admin/trips
/admin/shifts
/admin/vehicles
/admin/drivers
/admin/documents
/admin/reports
```

### Правило
Роутинг должен проверять роль пользователя и не пускать его в чужую зону интерфейса.

---

## 9. Модели и сериализация

Используйте:
- `freezed`
- `json_serializable`

### Почему
- меньше ручных ошибок;
- immutable-модели;
- аккуратная сериализация из Supabase JSON;
- легче держать порядок при росте таблиц и DTO.

---

## 10. Работа с Supabase

## 10.1. Что использовать напрямую
- auth;
- table queries;
- realtime subscriptions;
- storage upload/download;
- edge functions.

## 10.2. Что не делать
- не размазывать запросы по виджетам;
- не смешивать UI-модели и DB DTO в один тип;
- не делать весь код через raw maps без typed-моделей.

## 10.3. Рекомендация по репозиториям

Пример:
- `OrdersRepository`
- `TripsRepository`
- `ShiftsRepository`
- `VehiclesRepository`
- `DocumentsRepository`
- `ReportsRepository`

Каждый репозиторий:
- читает данные;
- пишет данные;
- отдает доменные модели;
- прячет детали запросов.

---

## 11. Работа с документами и storage

### Структура хранения

```text
documents/
  trips/{tripId}/ttn/
  trips/{tripId}/supporting/
  shifts/{shiftId}/
  expenses/{expenseId}/
  chat/{messageId}/
```

### Правила
- имена файлов генерировать серверно или через UUID;
- оригинал сохранять всегда;
- превью и сжатые версии можно генерировать позже;
- доступ к файлам должен контролироваться политиками, а не открытыми публичными URL.

---

## 12. Реaltime и чат

### Realtime использовать для
- новых сообщений;
- обновления статусов заявок;
- новых уведомлений;
- появления новых рейсов в журналах.

### Не пытаться сделать на старте
- сложную офлайн-синхронизацию чата;
- delivery/read receipts уровня мессенджера;
- реакцию на сообщения и прочие необязательные социальные функции.

---

## 13. Edge Functions

Edge Functions логично использовать для:
- генерации Excel-отчетов;
- вызова OCR;
- отправки push-уведомлений;
- генерации системных уведомлений;
- серверной нумерации заявок;
- фоновой проверки TO по дате/пробегу.

### Это важно
Не тяните бизнес-критичную логику нумерации и расчетов только в клиент. Ключевые операции должны иметь серверную опору.

---

## 14. Web-часть для руководителя

Flutter Web подходит, если:
- нужен один стек;
- интерфейс в основном табличный и формовый;
- вы готовы аккуратно проектировать responsive UI.

### Что учесть
- таблицы должны быть адаптированы под широкие экраны;
- фильтры и боковые панели не должны ломать мобильный вид;
- крупные модули админки лучше делать с layout-сеткой и постоянной навигацией.

## 16. Конфигурация окружений

Нужно минимум 3 окружения:
- `dev`
- `stage`
- `prod`

### Хранить отдельно
- Supabase URL;
- Supabase anon key;
- Sentry DSN;
- Yandex API keys;
- Firebase config.

### Правило
Секреты не должны жить в коде и коммититься в репозиторий в открытом виде.

---

## 17. Тестирование

Для соло-разработки реалистичный минимум:

### Unit tests
- расчеты зарплаты;
- проверки статусов;
- валидации смен;
- мапперы DTO.

### Widget tests
- формы входа;
- форма заявки;
- форма рейсового отчета;
- форма сменного отчета.

### Integration/manual tests
- авторизация;
- загрузка фото;
- OCR-цепочка;
- формирование отчета;
- push-уведомления.

Не надо обещать себе высокий coverage ради галочки. Нужны тесты на действительно рискованные места.

---

## 18. Логирование и мониторинг

### Подключить сразу
- Sentry для crash/error monitoring;
- базовые audit logs в БД;
- клиентские логи только для разработки.

### Ловить особенно внимательно
- падения при загрузке фото;
- ошибки OCR;
- ошибки генерации отчетов;
- проблемы авторизации;
- ошибки прав доступа.

---

## 19. Релизная стратегия

### Этап 1
- Android для водителей;
- Flutter Web для руководителя.

### Этап 2
- полировка web-кабинета;
- iOS при реальной необходимости;
- расширенная аналитика.

Это лучше, чем распыляться с первого дня на все платформы одинаково.

---

## 20. Предлагаемая последовательность технического старта

1. Создать репозиторий.
2. Поднять Flutter-проект.
3. Поднять Supabase-проект.
4. Создать auth и users/roles.
5. Подключить router, theme, Riverpod.
6. Сделать shell для driver/admin.
7. Реализовать справочники и машины.
8. Потом переходить к заявкам.
