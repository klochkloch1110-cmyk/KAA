# AVL 84 - ТЗ по экранам и сценариям v1

Документ описывает экраны системы и ключевые пользовательские сценарии для двух основных ролей:
- водитель;
- руководитель/админ.

Опциональная роль оператора в интерфейсах админа наследует большинство экранов руководителя, но с урезанными правами.

---

## 1. Общие UX-принципы

### Для водителя
- минимум шагов до действия;
- крупные кнопки и понятные статусы;
- акцент на текущую работу, а не на архив;
- минимум ручного ввода;
- быстрый доступ к камере и отправке отчета.

### Для руководителя
- быстрые фильтры и таблицы;
- много данных на экране без визуального мусора;
- удобная работа с документами и статусами;
- приоритет на контроль, поиск, выгрузку и аналитику.

## 3. Водительские экраны

## 3.1. Экран входа

### Назначение
Дать водителю простой и быстрый вход в систему.

### Состав экрана
- логотип/название компании;
- поле логина;
- поле пароля;
- кнопка "Войти";
- ссылка "Забыли пароль";
- индикатор загрузки;
- сообщение об ошибке.

### Действия
- ввести логин и пароль;
- войти;
- запросить восстановление доступа.

### Валидации
- логин обязателен;
- пароль обязателен;
- при ошибке авторизации показывать понятное сообщение без технического мусора.

### Состояния
- обычное;
- загрузка;
- ошибка входа;
- заблокированный пользователь.

---

## 3.2. Главный экран водителя

### Назначение
Показать, что у водителя происходит прямо сейчас.

### Основные блоки
- приветствие и имя;
- статус текущей смены;
- назначенная машина;
- активные заявки;
- быстрые действия;
- сводка за сегодня;
- последние уведомления.

### Быстрые действия
- открыть заявку;
- создать отчет по рейсу;
- закрыть смену;
- открыть машину;
- перейти в чат.

### Сводка за сегодня
- количество рейсов;
- суммарный объем;
- разбивка по заявкам;
- список уже отправленных ТТН;
- предварительный заработок за смену.

### Состояния
- есть активная смена;
- нет открытой смены;
- нет назначенных заявок;
- есть несданные отчеты;
- есть OCR-замечания от руководителя.

---

## 3.3. Список моих заявок

### Назначение
Показать водителю только его рабочие заявки.

### Элементы списка
- номер заявки;
- заказчик;
- маршрут A-B;
- материал;
- ставка за рейс;
- статус;
- количество уже выполненных рейсов по заявке.

Важно: водитель может видеть одну или несколько активных заявок в рамках смены. Ставка за перевезенную единицу для управленческого расчета водителю не показывается.

### Фильтры
- активные;
- завершенные;
- архив;
- по дате.

### Действия
- открыть карточку заявки;
- быстро открыть маршрут;
- начать отчет по рейсу.

---

## 3.4. Карточка заявки водителя

### Назначение
Показать все данные, нужные для выполнения работы.

### Блоки
- номер и дата заявки;
- заказчик;
- от кого / кому;
- материал;
- общий объем;
- ставка за рейс;
- примечание;
- погрузка;
- разгрузка;
- маршрут;
- назначенная машина;
- история рейсов по этой заявке.

### Кнопки
- открыть маршрут;
- создать рейсовый отчет;
- открыть чат;
- позвонить контактному лицу, если предусмотрено.

### UX-особенности
- точки A/B должны быть наглядны;
- открытие маршрута должно работать в один тап;
- история уже отправленных рейсов нужна, чтобы водитель видел, что он уже сдал.
- если в смене несколько заявок, водитель закрывает рейсы отдельно по каждой заявке.

## 3.5. Экран создания рейсового отчета

### Назначение
Позволить быстро отчитаться после каждого рейса.

### Поля формы
- дата и номер заявки - автозаполнение;
- ФИО водителя - автозаполнение;
- заказчик - автозаполнение;
- от кого - автозаполнение;
- кому - автозаполнение;
- номер ТТН - ручной ввод;
- объем перевезенного груза - ручной ввод;
- фото ТТН - обязательно;
- фото сопроводительных документов - необязательно;
- комментарий - необязательно.

### Кнопки
- сделать фото;
- загрузить из галереи;
- отправить отчет;
- сохранить черновик.

### Валидации
- нельзя отправить отчет без ТТН;
- номер ТТН обязателен;
- объем обязателен;
- если нет сети, можно сохранить как локальный черновик, если такая функция будет включена;
- OCR не блокирует отправку.

### После отправки
- показывается статус: отправлено, идет OCR-проверка;
- рейс появляется в списке текущей смены.

## 3.6. Список рейсов текущей смены

### Назначение
Показать водителю все рейсы в рамках смены и состояние отчетности.

### Содержимое карточки рейса
- время;
- номер заявки;
- маршрут;
- номер ТТН;
- объем;
- статус отправки;
- статус OCR.

### Статусы для отображения
- отправлен;
- требует сверки;
- подтвержден;
- отклонен/нужна корректировка.

---

## 3.7. Экран сменного отчета / закрытия смены

### Назначение
Завершить смену и собрать итоговые данные за день.

### Поля
- дата - автоматически;
- ФИО - автоматически;
- количество рейсов - автоматически;
- разбивка по заявкам - автоматически;
- общий объем - автоматически;
- номер машины - автоматически;
- пробег машины на конец смены - ручной ввод;
- заправлено ДТ - ручной ввод;
- комментарий по смене - необязательно.

### Кнопки
- отправить сменный отчет;
- сохранить черновик;
- вернуться к рейсам.

### Валидации
- пробег обязателен;
- пробег не может быть меньше последнего известного пробега машины;
- если в смене есть незавершенные рейсы или пустые обязательные поля, система предупреждает.

---

## 3.8. Экран назначенной машины

### Назначение
Показать водителю актуальную информацию по машине.

### Блоки
- марка и модель;
- госномер;
- текущий известный пробег;
- дата последнего ТО;
- последнее назначение;
- краткая история заправок и замечаний.

### Действия
- добавить замечание по машине, если функция будет включена;
- перейти к сменному отчету;
- открыть историю машины только в разрешенном объеме.

---

## 3.9. Чат водителя

### Назначение
Рабочее общение без выхода во внешний мессенджер.

### Состав
- лента сообщений;
- поле ввода;
- кнопка отправки;
- вложение фото;
- ответ на сообщение;
- системные сообщения по заявкам.

### Ограничения
- без сложных каналов на старте;
- без аудио и звонков на старте;
- приоритет скорости, стабильности и читаемости.

## 5. Сквозные пользовательские сценарии

## Сценарий 1. Руководитель создает заявку и назначает водителя
1. Руководитель открывает список заявок.
2. Нажимает "Создать заявку".
3. Заполняет данные.
4. Указывает машину и водителя.
5. Сохраняет заявку.
6. Водителю приходит уведомление.
7. Заявка появляется на главном экране водителя.

## Сценарий 2. Водитель выполняет рейс и сдает ТТН
1. Водитель открывает карточку заявки.
2. После рейса нажимает "Создать отчет по рейсу".
3. Вводит ТТН и объем.
4. Прикрепляет фото.
5. Отправляет отчет.
6. Система запускает OCR-проверку.
7. Руководитель видит рейс в журнале.

## Сценарий 3. Водитель закрывает смену
1. Водитель открывает экран смены.
2. Видит автоматический итог по рейсам.
3. Вводит пробег и заправку.
4. Отправляет сменный отчет.
5. Руководитель получает данные в журнале смен.

## Сценарий 4. Руководитель проверяет спорный OCR-документ
1. Руководитель открывает очередь OCR.
2. Видит расхождение по ТТН или объему.
3. Открывает карточку рейса и фото.
4. Принимает ручной ввод или корректирует.
5. Помечает рейс как проверенный.

## Сценарий 5. Руководитель формирует отчет за период
1. Открывает раздел отчетов.
2. Выбирает тип отчета и период.
3. Устанавливает фильтры.
4. Просматривает данные на экране.
5. Выгружает Excel-файл.

- просмотр документа fullscreen;
- фильтры в админке;
- мобильный и desktop-вариант ключевых экранов руководителя.
