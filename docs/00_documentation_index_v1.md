## Что уже можно делать в Figma

После этих документов уже можно переходить к UX-прототипированию.

Правильный порядок для Figma такой:

1. Сначала собрать user flow.
2. Потом сделать low-fidelity wireframes.
3. Потом утвердить структуру экранов.
4. Только после этого делать визуальный UI.

Если начать сразу с красоты, без утвержденных сценариев и состояний, вы получите красивый макет, который потом придется переделывать под реальную логику.

## Что еще важно учесть до начала дизайна

### 1. Проектировать не только идеальные состояния
Нужно сразу заложить:
- пустые состояния;
- ошибки сети;
- ошибки загрузки фото;
- отсутствие распознавания OCR;
- конфликт данных;
- отсутствие назначенной машины;
- закрытую смену;
- архивную или отмененную заявку.

### 2. Сразу думать о responsive-поведении админки
Кабинет руководителя должен нормально раскладываться в трех вариантах:
- телефон;
- планшет;
- ноутбук/монитор.

### 3. Не проектировать чат как отдельный мессенджер-продукт
Его задача - поддерживать операционную работу, а не соревноваться с Telegram.

### 4. Не перегружать экран водителя
У водителя интерфейс должен быть простым:
- что назначено;
- что нужно сделать сейчас;
- где отчитаться;
- где посмотреть свою смену, рейсы и заработок.

## Критерий готовности документации к старту разработки

Можно переходить к поднятию проекта и кода, когда:
- утверждены все экраны;
- утверждены поля всех форм;
- утверждены статусы;
- утверждены сущности БД;
- понятны роли и права;
- принято решение по стеку;
- Figma-прототипы не противоречат бизнес-правилам.

## Что логично делать следующим шагом после Figma и планирования

1. Поднять Supabase-проект.
2. Применить миграции из `supabase/migrations/`.
3. Создать тестовых пользователей в Supabase Auth.
4. Заполнить dev seed-данные.
5. Проверить RLS для ролей `admin` и `driver`.
6. Создать Flutter-проект в `app/`.
7. Реализовать auth, роли, shell админа и shell водителя.
8. Затем переходить к справочникам, машинам, заявкам, рейсам и сменам.

Ключевой ориентир: сначала рабочий сквозной сценарий `заявка -> рейс -> фото ТТН -> смена -> выгрузка`, потом OCR, чат, зарплата, расходы и AI.
