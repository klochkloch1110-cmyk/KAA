# AVL 84 - API, события и серверные операции v1

Документ фиксирует, какие операции должны быть клиентскими, какие серверными, и какие события нужны системе.

---

## 1. Общий подход

Клиент Flutter обращается к Supabase напрямую для обычных CRUD-операций, но бизнес-критичные действия выносятся в серверную сторону через SQL-функции или Edge Functions.

Критичные операции нельзя оставлять только на клиенте:

- генерация номера заявки;
- генерация отчетов;
- вызов OCR;
- отправка push-уведомлений;
- критичные пересчеты;
- массовые действия с документами.

---

## 2. Клиентские операции через Supabase SDK

### Auth
- вход;
- выход;
- получение текущего пользователя;
- восстановление пароля позже.

### Чтение данных
- профиль пользователя;
- свои заявки водителя;
- список заявок админа;
- машины;
- водители;
- справочники;
- рейсы;
- смены;
- документы;
- уведомления.

### Запись данных
- создание/редактирование справочников админом;
- создание/редактирование машин админом;
- создание/редактирование водителей админом;
- создание рейсового отчета водителем;
- закрытие смены водителем;
- создание документов после загрузки файлов.

## 3. Серверные операции

## 3.1. `generate-order-number`

Назначение: выдать следующий номер заявки.

Формат на старте:

```text
AVL-0001/05/26
```

Правила:
- номер уникален;
- генерация выполняется серверно;
- ручная правка доступна только админу и логируется.

## 3.2. `generate-report`

Назначение: собрать отчет за период.

Первый тип отчета:
- рейсы за период.

Позже:
- смены;
- зарплата;
- расходы;
- клиентский отчет.

## 3.3. `run-ocr`

Назначение: отправить фото ТТН на распознавание.

Первый релиз:
- функция может быть заглушкой;
- `ocr_status` у рейса остается `not_required` или `pending`.

После подключения OCR:
- сохранить сырой текст;
- извлечь номер ТТН;
- извлечь объем при возможности;
- сравнить с ручным вводом;
- выставить `matched`, `mismatch` или `failed`.

## 3.4. `send-notification`

Назначение: отправка push-уведомления и/или создание записи в `notifications`.

Первый релиз:
- достаточно записей в `notifications` без push.

## 4. Системные события

События нужны для уведомлений, аудита и будущей автоматизации.

### Заявки
- `order.created`
- `order.assigned`
- `order.status_changed`
- `order.archived`

### Рейсы
- `trip.submitted`
- `trip.verified`
- `trip.needs_review`
- `trip.rejected`
- `trip.ocr_mismatch`

### Смены
- `shift.opened`
- `shift.submitted`
- `shift.approved`
- `shift.needs_review`

### Документы
- `document.uploaded`
- `document.deleted_soft`
- `document.ocr_processed`

### Машины
- `vehicle.created`
- `vehicle.status_changed`
- `vehicle.odometer_updated`
- `vehicle.service_due`

## 5. Realtime-подписки

### Водитель
- новые назначенные заявки;
- изменение статуса своих заявок;
- комментарии/уведомления;
- статус проверки своих рейсов.

### Админ
- новые рейсы;
- новые сменные отчеты;
- новые документы;
- OCR-расхождения;
- незакрытые смены.

---

## 6. Audit log

Обязательно логировать:

- изменение статуса заявки;
- ручную правку номера заявки;
- подтверждение/отклонение рейса;
- изменение суммы расхода;
- изменение начисления зарплаты;
- архивирование критичных сущностей;
- блокировку пользователя.

---

## 7. Ошибки и сообщения пользователю

Технические ошибки не показывать пользователю напрямую.

Примеры пользовательских сообщений:

- `Не удалось загрузить фото. Проверьте интернет и попробуйте еще раз.`
- `Пробег меньше последнего сохраненного значения.`
- `Заявка уже отменена, по ней нельзя отправить рейс.`
- `Документ отправлен, OCR-проверка будет выполнена позже.`
